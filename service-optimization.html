服务履约系统 · 架构优化方案

PROBLEM ANALYSIS & SOLUTION DESIGN

现存核心问题
PROBLEM 01 · 性能
大 JSON 字段读取性能劣化
申请主表存储完整用户服务信息 JSON，读取时需加载整行数据、内存占用高；字段解析依赖应用层反序列化，无法按需取值；随数据体积增长，IO 与解析开销持续放大。
PROBLEM 02 · 状态机
多类型申请单状态机碎片化
各服务类型自持独立状态流转逻辑，既无法复用公共状态，也难以扩展个性化节点，代码维护成本极高。
PROBLEM 03 · 集成
三方供应商交互缺乏统一入口
供应商对接逻辑分散在各业务模块，鉴权、重试、限流、监控各自为战，变更一家供应商影响面难以评估。

优化方案 · 逐步拆解
Solution 01 · 存储拆分
大 JSON 拆分 + 冷热数据分离
📋 现状：申请主表承载过多职责
表存储内容问题
申请主表 用户基本信息 + 申请服务信息 + 申请履约信息 + 各类型附加字段（全部堆进大 JSON） 整行读取内存高 · 解析开销大 · 无法按需取值
履约历史状态表 状态变更流水记录 结构基本合理，可复用
图片证明材料表 服务申请履约相关图片附件 已独立，存储职责清晰
三方交互记录表 与供应商状态交互 + 回传信息 与供应商网关强耦合，归属模糊
↓ 拆分目标
🎯 目标：五表分责，冷热隔离
STEP 1
申请主表瘦身 → 只保留核心高频字段
将真正需要按需读取的核心字段保留在主表：申请ID、用户ID、服务类型、申请状态、创建时间、更新时间、当前受理供应商ID。其余字段全部迁出，单行数据体积可降低 70% 以上，每次读取的 IO 量与内存占用同比下降，列表场景无需加载任何冗余 JSON 内容。
STEP 2
新增「申请详情扩展表」→ 承接原 JSON 中的用户信息 & 服务信息
将用户基本信息、申请服务信息、申请履约信息迁入扩展详情表，与主表 1:1 关联。该表无需高频查询，允许保留 JSON 列存储非结构化细节，按需懒加载，不参与列表检索。
STEP 3
新增「服务类型扩展字段表」→ 消灭"适用部分类型"的冗余字段
各服务类型专属附加字段以 service_type + apply_id 为联合主键单独存储，主表不再出现大量 NULL 列。新增服务类型只需新增一张扩展表，主表结构零变更。
STEP 4
履约历史状态表 → 保留并强化索引
现有表结构职责清晰，建议在 apply_id + created_time 上补充联合索引，支撑状态时间线快速查询；同时增加 operator_type（系统/人工/供应商） 字段，为后续统一状态机审计做铺垫。
STEP 5
三方交互记录表 → 归并至供应商网关模块统一管理
该表的业务归属应迁移至 Solution 03 的供应商网关层，申请模块通过外键 + 事件引用关联，实现模块解耦。申请主表只保留「最新三方回传状态快照」字段，避免跨模块 JOIN。
5 张清晰职责独立表
主表字段 ↓70%列表查询全走索引
零 NULL 冗余类型扩展无主表变更
按需加载详情 / 附件 / 三方数据
Solution 02 · 状态机
统一状态机骨架 + 类型差异化扩展
⚠️ 根因：各服务类型"各自为政"，四大症结
📦
特药 / 绿通 / 护理 / 采血各写一套状态枚举，无共同语言，横向统计对比无法做
🔀
相同"处理中"语义，不同类型用了不同字符串，代码里到处 if-else 判断服务类型
🔌
供应商回调直接写死内部状态值，换一家供应商需大面积改业务代码
👤
用户操作 / 系统定时 / 三方回调 / 人工干预四路来源混用同一套逻辑，职责完全不清

🏗️ STEP 4 · 代码层：策略模式 + 状态模式 四层拆分
① StateMachineEngine
核心引擎（唯一入口）
接收 StateChangeEvent，完成幂等 → 规则校验 → 执行 transition → 写历史 → 发布事件的完整流程。所有触发源都调用此入口，不允许绕过。
② TransitionConfigLoader
流转规则加载器
从 DB 流转规则配置表加载：service_type + from_status + event → to_status + guard条件 + hook列表。规则热更新，无需重启服务即可调整流转逻辑。
③ ServiceTypeHandler<T>
类型策略接口（策略模式）
每种服务类型（特药/绿通/护理/采血）各实现一个 Handler，注册该类型的子状态序列与差异化前后置钩子。引擎按 service_type 路由分发，互不干扰。
④ StateEventListener
后置异步监听（观察者模式）
监听引擎发布的领域事件，异步处理：消息推送、供应商通知、SLA 计时、BI 上报等。与主流程完全解耦，新增后置动作无需修改引擎代码。
新增服务类型只需：实现 ServiceTypeHandler 接口 + 在配置表中录入流转规则，引擎代码零改动
统一骨架全类型状态横向可比
扩展零侵入新增类型不改引擎
四源统一入口杜绝绕过校验直改状态
规则热更新流转逻辑配置化可调
完整审计链每次流转可溯源回放
Solution 03 · 供应商网关
统一供应商接入网关
⚠️ 根因：接一家供应商 = 从头造一遍轮子
🔐
鉴权/签名：每家格式不同（HMAC / OAuth / 自定义），各写各的，散落在业务代码里
📤
推单接口：字段名、数据格式、协议（REST/SOAP）各异，转换逻辑重复堆叠
📥
回调接收：每家回调地址不同、格式不同，各自解析、各自验签，没有统一入口
🗺️
状态码映射：供应商返回自己的状态码，每次都要手写 Map 映射到内部骨架状态
♻️
重试/幂等/限流：超时重试、防重复回调、QPS 限制，每家各写一遍，质量参差不齐
📊
监控盲区：供应商调用耗时、失败率、回调延迟无统一观测，出问题靠日志捞针

📌 网关侧的变化
网关不再需要路由决策逻辑，入参直接携带 vendor_code，网关根据此字段加载对应 Adapter 及其配置，分发调用。
📌 同类型不同供应商怎么处理
同一服务类型可存在多个服务包，每个服务包绑定不同供应商。用户持 A 渠道的权益卡走供应商 X，持 B 渠道的走供应商 Y，决策在激活时完成，与网关无关。
📌 新增供应商的影响面
渠道侧：在服务包中配置新供应商 ID。网关侧：新增一个 Adapter + 录入供应商基础配置及状态映射配置。业务代码和状态机：零改动。
⚙️ STEP 5 · 通用基础能力 & 配置化管理
🔐 鉴权 & 签名
各家鉴权方式（HMAC/OAuth2/API-KEY）封装为 AuthStrategy，按供应商加载，网关核心层统一调用
♻️ 重试 & 熔断
指数退避重试（可配置次数/间隔），失败率超阈值自动熔断，熔断期间直接走备用供应商
🔒 幂等控制
推单以 apply_id + vendor_code 为幂等 key 防重，回调以供应商流水号去重，Redis 存储幂等状态
📊 可观测性
每次调用记录耗时、状态码、回调延迟，暴露 Metrics 接口，统一监控看板实时观测各供应商健康状态

配置元数据层 · 履约项防膨胀设计
⚠️ 履约项膨胀的三个根因
01
跨服务类型高度相似的履约项（姓名、手机号等）各自独立定义，没有复用
02
同一履约项在不同服务包/渠道下校验规则略有差异，衍生出大量相似变体
03
不同渠道/服务包为细微定制需求各自新建一次性专属履约项，长期累积无人清理
✅ 防膨胀三板斧
🧱
原子化 + 复用机制
履约项定义与业务场景解耦，抽象为原子履约项（字段类型 + 基础校验），动态表单引用而非拷贝。同一"手机号"履约项可被特药、绿通、护理共用，修改一次全局生效。

履约项只定义字段本质（类型、展示名），校验规则作为独立配置层附加在「动态表单引用关系」上，同一履约项可在不同表单里配置不同必填/正则/范围规则，无需为细微差异新建履约项变体。

🔗 四层配置依赖关系总览
Layer 0
履约项
field_key · 字段类型 · 展示名 · 引用计数 · 状态(启用/归档)
← 原子单元，跨场景复用，校验规则不在此定义
↓ 引用（含表单内校验规则配置）
Layer 1
动态表单
form_id · 表单名 · [履约项引用列表 + 各自校验规则 + 排序 + 是否必填]
← 驱动前端渲染，同一履约项在不同表单可有不同校验
↓ 关联
Layer 2
服务项
service_item_id · 服务名称 · 服务类型 · 关联 form_id · 服务描述
← 定义一个可被用户申请的具体服务
↓ 打包
Layer 3
服务包
package_id · 包名 · [服务项列表] · vendor_code · 渠道归属 · 有效期
← 供应商在此绑定，渠道在此层配置给用户
